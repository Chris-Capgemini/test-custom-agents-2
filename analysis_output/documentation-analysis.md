## 1. Executive Summary

The Allegro PoC is a **three-component polyglot system** (Java Swing + Node.js + Vue.js) with **794 lines of code across 15 source files**, implementing a real-time person-data transfer workflow from a browser to a legacy desktop application via WebSocket. Its documentation, however, tells almost none of this story.

**The fundamental problem**: the root README presents the project as a simple Java Swing setup guide with a Docker dependency. The Node.js WebSocket relay server — the architectural backbone connecting all three components — has zero user-facing documentation. Two of the six documentation files contain hardcoded personal Windows file paths from the original developer (`C:/Users/esultano/...`), making them actively misleading for any other user.

## 5. Critical Gaps

> These gaps make it impossible for a new developer to understand, set up, or use the system.

### CG-01 — No System Overview or Project Purpose Statement

**Impact**: A new developer reading the repository sees a "Java Swing Project Setup Guide". There is no explanation of what ALLEGRO is, what the PoC demonstrates, why there are three components, or what the German field names mean.

**Recommendation**: Add a project overview section at the top of `README.md`:
```markdown
## About This Project
Allegro PoC demonstrates modernisation of a legacy enterprise desktop application 
(ALLEGRO) by bridging a Java Swing desktop form with a modern Vue.js browser UI 
via a Node.js WebSocket relay. Users can search for a person in the browser and 
transfer their data to the Allegro desktop form in real time.
```

---

### CG-02 — Node.js WebSocket Relay Has Zero Documentation

**Impact**: The middleware component connecting browser to desktop is entirely invisible. Without it, neither the Vue client nor the WebSocket sync features can function. A developer reading current documentation would not know it exists.

**What the code actually does** (from `WebsocketServer.js` analysis):
- Listens on port `1337` for WebSocket connections
- Broadcasts any received message to **all** connected clients
- Handles HTTP upgrade requests for WebSocket handshake
- Supports multiple simultaneous client connections

**Recommendation**: Create `node-server/README.md`:
```markdown
# WebSocket Relay Server
Broadcast relay connecting the Vue.js browser client to the Java Swing desktop.

## Start
cd node-server && npm install && node src/WebsocketServer.js

## Port: 1337
## Message format: {"target": "<field-name>", "content": "<value>"}
```

---

### CG-03 — WebSocket API Completely Undocumented

**Impact**: `ws://localhost:1337` is the primary communication channel for the system's key features. It is not mentioned in any documentation file.

**What is implemented but undocumented**:
- Connection: `ws://localhost:1337`
- Inbound message from Vue client: `{ "target": "<ModelProperties key>", "content": "<value>" }`
- Broadcast: server relays message to all connected clients (including the Swing app)
- The Swing app's WebSocket client receives the broadcast and updates form fields

**Recommendation**: Add to `api.yml` a WebSocket API section, or create `api-websocket.md`.

---

### CG-04 — No End-to-End Startup Guide

**Impact**: Starting the system requires reading 4 different files, two of which have non-portable paths. The correct startup order is:
1. Start httpbin Docker container (port 8080)
2. Start Node.js WebSocket server (port 1337)
3. Start Vue.js dev server (port 8081 default)
4. Run Java Swing application

This sequence exists nowhere in the documentation.

---

### CG-05 — Person Search and Allegro Transfer Feature Not Documented

**Impact**: The primary user-facing business workflow — searching for a person in the Vue browser UI and clicking "Transfer to Allegro" to populate the Swing form via WebSocket — is never mentioned.

**What the code implements** (from `Search.vue` analysis):
- Search input filters a mock dataset of 5 persons
- Person data includes: Vorname, Name, Adresse, IBAN, BIC, Kontonummer fields
- "Übertragen" (Transfer) button sends each field via WebSocket message to the relay
- Relay broadcasts to Swing app which populates corresponding form fields
