## 📌 Overview

This repository is a **Proof-of-Concept (PoC)** for the modernisation of the **Allegro** legacy system. It demonstrates a WebSocket-based integration between three distinct tiers:

1. **Java Swing Desktop Client** — a German-language customer data entry form built with MVP architecture
2. **Node.js WebSocket Relay Server** — a broadcast hub connecting web and desktop clients
3. **Vue.js Web Frontend** — a browser-based search interface that sends customer data to the Swing client

The application flow is:  
`Vue.js Search UI` → `(WebSocket)` → `Node.js Relay Server` → `(WebSocket)` → `Java Swing Client` → `(HTTP POST)` → `HTTPBin Echo Service`

The primary business scenario is: **search for a customer in the web UI, select them and their payment details, and transfer that data directly into the Allegro Swing desktop application.**

## 🏢 Key Business Capabilities Identified

### 1. 🔍 Customer Search (Vue.js)
- **File:** `Search.vue`
- Multi-field OR search across: last name, first name, ZIP (exact), city, street, house number
- Searches a mock in-memory dataset of 5 German customer records
- Case-insensitive substring matching for all text fields

### 2. 💳 Payment Recipient Selection (Vue.js)
- **File:** `Search.vue`
- Each customer has 1–3 Zahlungsempfänger (payment recipients) with IBAN, BIC, Gültig-ab
- User selects one payment record; the selected entry replaces the array before WebSocket transfer

### 3. 📤 Allegro Data Transfer (Vue.js → Node.js → Java)
- **File:** `Search.vue`, `WebsocketServer.js`, `websocket/Main.java`
- "Nach ALLEGRO übernehmen" button sends person + selected payment over WebSocket
- Node.js relay broadcasts to all connected clients
- Swing client parses JSON and populates all form fields

### 4. 📝 Real-Time Text Area Sync (Vue.js → Node.js → Java)
- **File:** `Search.vue`, `websocket/Main.java`
- Vue watcher on textarea sends content live to Swing client on every keystroke
- Target `'textarea'` routes to the text area component in the Swing UI

### 5. 📮 HTTP Form Submission (Java Swing)
- **File:** `PocModel.java`, `HttpBinService.java`
- "Anordnen" button in Swing client POSTs all 13 form fields to HTTPBin echo endpoint
- Response body is displayed back in the Swing text area

## 🌐 WebSocket Message Protocol

The application uses a shared JSON envelope for all WebSocket messages:

```json

  "target": "textfield | textarea",
  "content": "<object_or_string>"
}
```

## ⚠️ Notable Findings & Technical Debt

### 🔴 High Priority
1. **Null safety risk in PocModel.action()** — `model.get(val).getField().toString()` will throw `NullPointerException` if any field is null (all are initialised to null). No null guard exists.
2. **No input validation** — Neither the Swing MVP nor the Vue search form validates any input before submission.
3. **Hardcoded localhost URLs** — WebSocket endpoint (`ws://localhost:1337/`) and HTTP endpoint (`http://localhost:8080/post`) are hardcoded throughout, with no configuration mechanism.

### 🟡 Medium Priority
4. **No WebSocket reconnection** — Both Java and Vue clients make no attempt to reconnect on disconnect.
5. **No authentication or origin validation** — The Node.js server accepts connections from any origin (`null` check).
6. **messages array declared but never used** in `WebsocketServer.js`.
7. **ViewData.java is empty** — scaffolded placeholder with no implementation.
8. **Dual Main.java implementations** — `websocket/Main.java` (legacy monolithic) and `com/Main.java` (MVP-based) both exist; the legacy version appears superseded but not removed.

### 🟢 Low Priority
9. **Gender flags as independent booleans** — FEMALE/MALE/DIVERSE are three separate model properties rather than a single enum, creating potential for invalid states where multiple are true simultaneously.
10. **HTTP error handling gap** — `HttpBinService` has no `finally` block to disconnect on error, and `PocModel.action()` only checks for empty body, not HTTP status codes.
