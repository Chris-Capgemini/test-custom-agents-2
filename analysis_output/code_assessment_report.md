### ISS-001: NullPointerException Crash on Every Submit Press
- **File:** `swing/src/main/java/com/poc/model/PocModel.java` — Line 39
- **Type:** Reliability
- **Estimated Fix Time:** 1 hour

**Problem:**
```java
// PocModel.java line 39 — GUARANTEED NullPointerException
data.put(val.toString(), model.get(val).getField().toString());
//                                              ^^^^^^^^
// All 13 fields are initialised to new ValueModel<String>(null) at lines 17–29.
// getField() returns null. Calling .toString() on null throws NPE.
```

Every field is initialised to `new ValueModel<String>(null)`. When the user presses "Anordnen" without completing the form, `getField()` returns `null` and `.toString()` crashes immediately. The application is non-functional for its primary submit action from first launch.

**Solution:** Replace the raw `.toString()` call with a null-safe accessor. Use `Objects.toString(model.get(val).getField(), "")`. Additionally, add a `validateRequiredFields()` method to `PocModel` that enforces at minimum `FIRST_NAME`, `LAST_NAME`, and `IBAN` are non-empty before calling `action()`. Show a `JOptionPane` error dialog from `PocPresenter` when validation fails, so the user receives meaningful feedback rather than a crash.

```java
// Fix — PocModel.java line 39:
data.put(val.toString(),
    Objects.toString(model.get(val).getField(), ""));
```

### ISS-002: Blocking HTTP Call on the Swing Event Dispatch Thread
- **File:** `swing/src/main/java/com/poc/presentation/PocPresenter.java` — Lines 40–48
- **Type:** Performance / Reliability
- **Estimated Fix Time:** 2 hours

**Problem:**
```java
this.view.button.addActionListener(_ -> {
    try {
        model.action();    // ← synchronous HTTP POST blocks EDT entirely
    } catch (IOException e) {
        throw new RuntimeException(e);  // ← no user feedback, no logging
    }
});
```

`model.action()` calls `HttpBinService.post()` which is a blocking `HttpURLConnection` call executing directly on the Event Dispatch Thread. The **entire Swing UI freezes** for the duration of the network call. With no timeout configured, a downed HTTPBin service will hang the application permanently with no ability to cancel.

**Solution:** Wrap in `SwingWorker`. Move the HTTP call to `doInBackground()` (background thread), handle the result in `done()` (back on EDT). Disable the submit button during the call and restore it in `done()`. Display a loading cursor and show any error via `JOptionPane`.

```java
this.view.button.addActionListener(_ -> {
    view.button.setEnabled(false);
    view.frame.setCursor(Cursor.getPredefinedCursor(Cursor.WAIT_CURSOR));
    new SwingWorker<String, Void>() {
        @Override protected String doInBackground() throws Exception {
            return model.action();  // off-EDT
        }
        @Override protected void done() {
            view.button.setEnabled(true);
            view.frame.setCursor(Cursor.getDefaultCursor());
            try { get(); }
            catch (Exception e) {
                JOptionPane.showMessageDialog(view.frame,
                    "Übertragung fehlgeschlagen: " + e.getCause().getMessage(),
                    "Fehler", JOptionPane.ERROR_MESSAGE);
            }
        }
    }.execute();
});
```

### ISS-003: MVP Architecture Has No WebSocket Integration — Refactoring is Incomplete
- **File:** `swing/src/main/java/com/Main.java` (and entire `com/poc/*` package)
- **Type:** Business Logic
- **Estimated Fix Time:** 10 hours

**Problem:** The refactored MVP entry point (`com/Main.java`) creates `PocView`, `PocModel`, `PocPresenter`, and `EventEmitter` — but **never instantiates a WebSocket client**. There is no `@ClientEndpoint` class anywhere in the `com.poc.*` package. Form fields are therefore **never populated from WebSocket messages** in the MVP code path.

The working WebSocket message handling lives entirely in `websocket/Main.java` (the legacy 457-line monolith). The MVP refactoring migrated the HTTP submission path but left the WebSocket integration — the *primary* PoC capability — behind. Running `com.Main` produces a form that can only be filled manually; the whole-system scenario (Vue search → relay → auto-populate Swing form) is broken via the MVP entry point.

**Solution:** Create a `WebSocketService` class in a new `com.poc.ws` package with proper `@ClientEndpoint` lifecycle callbacks. Parse incoming JSON and call `model.setValue()` for each field. Instantiate it from `com.Main`. Migrate the `toSearchResult()` JSON parsing from `websocket/Main.java` as the implementation reference. Use `SwingUtilities.invokeLater()` for all model updates triggered from `@OnMessage`.

```java
// com/poc/ws/WebSocketService.java (new)
@ClientEndpoint
public class WebSocketService {
    private final PocModel model;
    private Session session;

    public WebSocketService(PocModel model) { this.model = model; }

    public void connect(URI uri) throws DeploymentException, IOException {
        ContainerProvider.getWebSocketContainer().connectToServer(this, uri);
    }

    @OnMessage
    public void onMessage(String json) {
        SwingUtilities.invokeLater(() -> {
            CustomerPayload p = CustomerPayloadParser.parse(json);
            model.setValue(ModelProperties.FIRST_NAME, p.getFirst());
            model.setValue(ModelProperties.LAST_NAME,  p.getName());
            model.setValue(ModelProperties.IBAN,       p.getIban());
            // ... all fields including HAUSNUMMER and KUNDENNUMMER
        });
    }
}

// com/Main.java addition:
var wsService = new WebSocketService(pocModel);
wsService.connect(URI.create(ApplicationConfig.getWebSocketUrl()));
```

### ISS-004: Resource Leak in HttpBinService — Connection Never Closed on Exception
- **File:** `swing/src/main/java/com/poc/model/HttpBinService.java` — Lines 16–36
- **Type:** Reliability
- **Estimated Fix Time:** 0.5 hours

`HttpURLConnection` and `Scanner` are never closed if any `IOException` occurs between `openConnection()` and `disconnect()`. Additionally, `Scanner.next()` throws `NoSuchElementException` on an empty response body — before `disconnect()` is reached — guaranteeing a permanent connection leak on that path.

```java
// Fix: try-finally + conditional read + timeout
try {
    connection.setConnectTimeout(5000);
    connection.setReadTimeout(10000);
    // write request body ...
    int code = connection.getResponseCode();
    if (code < 200 || code >= 300)
        throw new IOException("HTTP error: " + code);
    try (var scanner = new Scanner(connection.getInputStream()).useDelimiter("\\A")) {
        return scanner.hasNext() ? scanner.next() : "";
    }
} finally {
    connection.disconnect();
}
```

### ISS-007: JTextArea Added to Panel Twice — Layout Bug
- **File:** `swing/src/main/java/com/poc/presentation/PocView.java` — Lines 188–189

- **Estimated Fix Time:** 0.25 hours

```java
panel.add(textArea);      // ← line 188: added unconditionally (no constraints)
```

In Swing, a component can only have one parent. The second `add()` silently removes the component from its first position and re-adds it. ## Summary of Key Findings

The Allegro Modernisation PoC demonstrates a well-conceived three-tier architecture and a clean MVP decomposition in the Java layer. However, **three critical defects prevent the application from functioning correctly**:

1. **`PocModel.action()` crashes on every submit** with a `NullPointerException` — all 13 fields are initialised to `null` and `.toString()` is called on them unconditionally.
2. **The submit button freezes the entire Swing UI** because a blocking HTTP POST is executed directly on the Event Dispatch Thread with no timeout.
3. **The MVP entry point (`com.Main`) has no WebSocket client** — the primary PoC scenario (Vue frontend auto-populating the Swing form) is completely non-functional via the refactored code path.

Beyond these critical issues, **two data completeness regressions** from the refactoring (`HAUSNUMMER`, `KUNDENNUMMER` absent from the MVP model) mean that even when the WebSocket is integrated, two key business fields would be silently dropped. The estimated effort to bring this PoC to a fully-functional and testable state is approximately **23 hours** (immediate + short-term actions). Production readiness requires an additional **~80 hours** primarily driven by test coverage and API integration.
