## 2. Sequence Diagram — Person Search & WebSocket Transfer to Swing

### Description

Shows the complete flow when a user searches for a person in the **Vue.js browser client**, selects a result, and transfers the person's data to the **Java Swing Allegro form** via the **Node.js WebSocket relay server**.

Key steps:
1. Client-side search against in-memory `search_space[]` (no server roundtrip for search)
2. User selects person row + optional `Zahlungsempfänger` (payment info) row
3. `sendMessage()` wraps the payload as `{ target: "textfield", content: personData }` and sends over WebSocket
4. Node.js relay broadcasts to **all** connected clients
5. `WebsocketClientEndpoint.onMessage()` parses the JSON and dispatches field-by-field updates to the Swing `JTextField` components

```mermaid
sequenceDiagram
    actor User as 🧑 User (Browser)
    participant SearchVue  as Search.vue
    participant BrowserWS  as WebSocket (Browser)
    participant NodeServer as WebsocketServer.js (Node.js :1337)
    participant JavaWS     as WebsocketClientEndpoint (Java)
    participant SwingUI    as websocket/Main — Swing UI

    %% ── INITIAL CONNECTION ─────────────────────────────────
    Note over SearchVue, NodeServer: On mount: Search.vue calls connect() → new WebSocket("ws://localhost:1337/")
    SearchVue->>NodeServer: WebSocket CONNECT
    NodeServer-->>SearchVue: Connection accepted (clients[] push)

    Note over JavaWS, NodeServer: Java client already connected on startup
    JavaWS->>NodeServer: WebSocket CONNECT
    NodeServer-->>JavaWS: Connection accepted (WebsocketClientEndpoint.onOpen)

    %% ── PERSON SEARCH ──────────────────────────────────────
    rect rgb(227, 242, 253)
        Note over User, SearchVue: Step 1 — Person Search (client-side only)
        User->>SearchVue: Enter criteria (name, first, zip, ort, street, hausnr)
        User->>SearchVue: Click "Suchen" button
        activate SearchVue
        SearchVue->>SearchVue: searchPerson()<br/>filter search_space[] by criteria
        SearchVue-->>User: Render matching rows in results table
        deactivate SearchVue
    end

    %% ── RESULT SELECTION ───────────────────────────────────
    rect rgb(232, 245, 233)
        Note over User, SearchVue: Step 2 — Select Person & Payment Info
        User->>SearchVue: Click result row
        activate SearchVue
        SearchVue->>SearchVue: selectResult(item)<br/>selected_result = item
        SearchVue-->>User: Row highlighted (CSS class 'highlight')
        deactivate SearchVue

        opt User selects a Zahlungsempfänger (payment record)
            User->>SearchVue: Click Zahlungsempfänger row
            activate SearchVue
            SearchVue->>SearchVue: zahlungsempfaengerSelected(item)<br/>zahlungsempfaenger_selected = item
            SearchVue-->>User: Row highlighted (green)
            deactivate SearchVue
        end
    end

    %% ── TRANSFER TO ALLEGRO ────────────────────────────────
    rect rgb(243, 229, 245)
        Note over User, SwingUI: Step 3 — Transfer to Allegro via WebSocket
        User->>SearchVue: Click "Nach ALLEGRO übernehmen"
        activate SearchVue

        SearchVue->>SearchVue: sendMessage(selected_result, "textfield")<br/>Merge zahlungsempfaenger_selected into payload
        SearchVue->>BrowserWS: socket.send(JSON.stringify({<br/>  target: "textfield",<br/>  content: personData }))
        deactivate SearchVue

        activate BrowserWS
        BrowserWS->>NodeServer: WebSocket message frame (utf8)
        deactivate BrowserWS

        activate NodeServer
        Note over NodeServer: Received message — broadcast to ALL clients[]
        loop for each connected client
            NodeServer->>NodeServer: clients[i].sendUTF(json)
        end
        NodeServer->>JavaWS: sendUTF(json) → Java Swing client
        NodeServer->>BrowserWS: sendUTF(json) → Vue browser (echo)
        deactivate NodeServer
    end

    %% ── SWING FIELD POPULATION ─────────────────────────────
    rect rgb(232, 234, 246)
        Note over JavaWS, SwingUI: Step 4 — Parse & Populate Swing Form
        activate JavaWS
        JavaWS->>JavaWS: onMessage(json)
        JavaWS->>JavaWS: extract(json) → Message { target:"textfield", content:json }

        alt message.target == "textfield"
            JavaWS->>JavaWS: toSearchResult(message.content)<br/>→ SearchResult (JSON parsed field-by-field)
            JavaWS->>SwingUI: tf_name.setText(searchResult.name)
            JavaWS->>SwingUI: tf_first.setText(searchResult.first)
            JavaWS->>SwingUI: tf_dob.setText(searchResult.dob)
            JavaWS->>SwingUI: tf_zip.setText(searchResult.zip)
            JavaWS->>SwingUI: tf_ort.setText(searchResult.ort)
            JavaWS->>SwingUI: tf_street.setText(searchResult.street)
            JavaWS->>SwingUI: tf_hausnr.setText(searchResult.hausnr)
            JavaWS->>SwingUI: tf_ze_iban.setText(searchResult.ze_iban)
            JavaWS->>SwingUI: tf_ze_bic.setText(searchResult.ze_bic)
            JavaWS->>SwingUI: tf_ze_valid_from.setText(searchResult.ze_valid_from)
        else message.target == "textarea"
            JavaWS->>SwingUI: textArea.setText(message.content)
        end
        deactivate JavaWS

        SwingUI-->>User: ✅ Swing form populated with person data
    end
```

### Notes

- **No server-side search**: `searchPerson()` filters the hardcoded `search_space[]` array entirely in-browser
- **Broadcast model**: Node.js relays every message to **all** connected clients; both the browser tab and the Java Swing window receive the same JSON
- **JSON parsing**: `extract()` uses `javax.json` streaming API (not Jackson/Gson) to walk the JSON token stream
- **Textarea path**: If `target == "textarea"`, the content goes directly to `JTextArea.setText()` without `SearchResult` mapping

## 3. Sequence Diagram — Form Submission to HTTPBin Backend

### Description

Shows the MVP binding lifecycle and the full form-submission flow when the user clicks the **"Anordnen"** (Submit/Arrange) button in the `PocView` Swing form.

Key steps:
1. **Binding phase** (at startup): `PocPresenter.initializeBindings()` attaches `DocumentListener` to every `JTextComponent` and `ChangeListener` to every `JRadioButton` — each listener writes directly into the matching `ValueModel<T>` in `PocModel.model`
2. **Runtime**: Every keystroke or radio-button change silently updates the `EnumMap` entry
3. **Submission**: Button click triggers `model.action()` → iterates all 13 `ModelProperties` → builds `HashMap<String,String>` → HTTP POST via `HttpBinService`
4. **Response**: `EventEmitter.emit(responseBody)` fires; the registered lambda in `PocPresenter` clears the form and displays the response

```mermaid
sequenceDiagram
    actor User as 🧑 User
    participant PocView       as PocView (Swing)
    participant PocPresenter  as PocPresenter
    participant PocModel      as PocModel
    participant ValueModel    as ValueModel~T~ (×13 in EnumMap)
    participant HttpBinSvc    as HttpBinService
    participant HTTPBin       as HTTPBin Server (localhost:8080)
    participant EventEmitter  as EventEmitter
    participant Listener      as EventListener (lambda in Presenter)

    %% ── STARTUP: BINDING PHASE ─────────────────────────────
    rect rgb(255, 248, 225)
        Note over PocPresenter, ValueModel: Startup — initializeBindings() (runs once in constructor)
        PocPresenter->>PocPresenter: initializeBindings()
        loop bind() called 13× (10 JTextComponent + 3 JRadioButton)
            PocPresenter->>ValueModel: model.setField(source.getText() / isSelected())
            PocPresenter->>PocView: source.getDocument().addDocumentListener(...)
            Note right of PocPresenter: DocumentListener writes to<br/>ValueModel on every change
        end
        PocPresenter->>EventEmitter: subscribe(λ → update view on response)
        PocPresenter->>PocView: button.addActionListener(λ → model.action())
    end

    %% ── RUNTIME: LIVE FIELD UPDATES ────────────────────────
    rect rgb(227, 242, 253)
        Note over User, ValueModel: Runtime — live two-way binding (repeated for each field edit)
        User->>PocView: Type in text field / toggle radio button
        activate PocView
        PocView->>PocPresenter: DocumentEvent (insertUpdate / removeUpdate)
        activate PocPresenter
        PocPresenter->>ValueModel: setField(e.getDocument().getText(0, length))
        Note right of ValueModel: EnumMap entry updated silently
        deactivate PocPresenter
        deactivate PocView
    end

    %% ── SUBMISSION ─────────────────────────────────────────
    rect rgb(232, 245, 233)
        Note over User, HTTPBin: Form Submission — "Anordnen" button clicked
        User->>PocView: Click "Anordnen" button
        activate PocView
        PocView->>PocPresenter: ActionEvent fired
        activate PocPresenter
        PocPresenter->>PocModel: model.action()
        activate PocModel

        Note over PocModel: Iterate all 13 ModelProperties
        loop for each val in ModelProperties.values()
            PocModel->>ValueModel: model.get(val).getField()
            ValueModel-->>PocModel: current field value
            PocModel->>PocModel: data.put(val.toString(), value.toString())
        end

        PocModel->>HttpBinSvc: post(data: HashMap~String,String~)
        activate HttpBinSvc

        HttpBinSvc->>HttpBinSvc: Open HttpURLConnection to http://localhost:8080/post
        HttpBinSvc->>HttpBinSvc: setRequestMethod("POST")<br/>setRequestProperty("Content-Type","application/json")
        HttpBinSvc->>HttpBinSvc: Write JSON body via javax.json generator
        HttpBinSvc->>HTTPBin: HTTP POST /post (Content-Type: application/json)
        activate HTTPBin
        HTTPBin-->>HttpBinSvc: 200 OK + JSON responseBody
        deactivate HTTPBin

        HttpBinSvc->>HttpBinSvc: Read response via Scanner
        HttpBinSvc->>HttpBinSvc: connection.disconnect()
        HttpBinSvc-->>PocModel: responseBody: String
        deactivate HttpBinSvc
    end

    %% ── RESPONSE HANDLING ──────────────────────────────────
    rect rgb(243, 229, 245)
        Note over PocModel, Listener: Response Dispatch via Observer Pattern
        alt responseBody is NOT empty
            PocModel->>EventEmitter: emit(responseBody)
        else responseBody is empty
            PocModel->>EventEmitter: emit("Failed operation")
        end
        deactivate PocModel

        activate EventEmitter
        loop for each registered EventListener
            EventEmitter->>Listener: listener.onEvent(eventData)
        end
        deactivate EventEmitter

        activate Listener
        Note over Listener, PocView: Presenter's lambda resets the form
        Listener->>PocView: textArea.setText(eventData)
        Listener->>PocView: firstName.setText("")
        Listener->>PocView: name.setText("")
        Listener->>PocView: dateOfBirth.setText("")
        Listener->>PocView: zip.setText("")
        Listener->>PocView: ort.setText("")
        Listener->>PocView: street.setText("")
        Listener->>PocView: iban.setText("")
        Listener->>PocView: bic.setText("")
        Listener->>PocView: validFrom.setText("")
        Listener->>PocView: female.setSelected(true)<br/>male.setSelected(false)<br/>diverse.setSelected(false)
        deactivate Listener

        deactivate PocPresenter
        deactivate PocView
        PocView-->>User: ✅ Response displayed in textArea, form fields cleared
    end
```

### Notes

- **Binding is synchronous**: `initializeBindings()` runs inside the `PocPresenter` constructor; the form is live-bound before the user can interact
- **No validation**: `model.action()` submits all 13 fields as-is; calling `.toString()` on a `null` field causes a `NullPointerException` at runtime (known technical debt)
- **EventEmitter as integration bus**: Both success and error paths go through `EventEmitter`, keeping the Presenter decoupled from the HTTP layer
- **Thread safety**: The EventEmitter callback runs on the EDT if called from the button's ActionListener; HTTPBin call is blocking on the EDT (potential freeze — tech debt)
